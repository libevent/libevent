# 第 1 课：IO 多路复用原理

> **前置要求**：熟悉 C 语言、Linux socket 基础（socket/bind/listen/accept/read/write）
> **本课目标**：彻底理解 select / poll / epoll 的工作原理与局限，建立"为什么需要 libevent"的直觉

---

## 1. 问题的起点：一个服务器如何同时服务多个客户端？

假设我们要写一个 TCP echo 服务器，最朴素的写法是这样的：

```c
// 最简单的单客户端服务器（阻塞 IO）
int server_fd = socket(AF_INET, SOCK_STREAM, 0);
bind(server_fd, ...);
listen(server_fd, 128);

while (1) {
    int client_fd = accept(server_fd, NULL, NULL); // 阻塞，等待连接
    char buf[1024];
    int n;
    while ((n = read(client_fd, buf, sizeof(buf))) > 0) { // 阻塞，等待数据
        write(client_fd, buf, n);
    }
    close(client_fd);
}
```

**问题**：`read` 会阻塞——在等待客户端 A 发数据时，客户端 B 的连接请求就被晾在那里了。
这个服务器**同一时刻只能服务一个客户端**。

### 1.1 直觉上的解决方案：多进程 / 多线程

```c
// 多进程版本
while (1) {
    int client_fd = accept(server_fd, NULL, NULL);
    if (fork() == 0) {          // 子进程处理该客户端
        handle_client(client_fd);
        exit(0);
    }
    close(client_fd);           // 父进程关闭副本，继续 accept
}
```

这可以工作，但代价昂贵：
- 每个连接一个进程/线程，10000 个连接 = 10000 个线程
- 线程切换的 CPU 开销、栈内存占用（默认 8MB/线程）
- C10K 问题（2000 年代初的工程瓶颈）由此而来

### 1.2 根本思路：IO 多路复用（I/O Multiplexing）

核心思想：**用一个线程，同时监听多个 fd，哪个 fd 就绪就处理哪个**。

```
单线程同时监听 N 个 fd：

fd1 (client A) ────┐
fd2 (client B) ────┤
fd3 (client C) ────┼──→ [监听机制] → 就绪的 fd → 处理
...                │
fdN               ─┘
```

操作系统提供了三种实现这一思想的系统调用：**select**、**poll**、**epoll**。

## 2. select：第一代多路复用

### 2.1 接口

```c
#include <sys/select.h>

int select(int nfds,
           fd_set *readfds,
           fd_set *writefds,
           fd_set *exceptfds,
           struct timeval *timeout);

// fd_set 操作宏
FD_ZERO(&set);          // 清空集合
FD_SET(fd, &set);       // 将 fd 加入集合
FD_CLR(fd, &set);       // 从集合移除 fd
FD_ISSET(fd, &set);     // 检查 fd 是否在集合中（结果判断）
```

### 2.2 工作流程

```c
// 用 select 监听多个 fd 的模板代码
fd_set read_fds;
int maxfd = 0;

// --- 注册阶段 ---
FD_ZERO(&read_fds);
FD_SET(server_fd, &read_fds);
maxfd = server_fd;
for (int i = 0; i < n_clients; i++) {
    FD_SET(client_fds[i], &read_fds);
    if (client_fds[i] > maxfd) maxfd = client_fds[i];
}

// --- 等待阶段 ---
struct timeval tv = {5, 0};  // 最多等 5 秒
int nready = select(maxfd + 1, &read_fds, NULL, NULL, &tv);

// --- 检查阶段（线性扫描）---
if (FD_ISSET(server_fd, &read_fds)) {
    // server_fd 可读 → 有新连接
    accept(server_fd, ...);
}
for (int i = 0; i < n_clients; i++) {
    if (FD_ISSET(client_fds[i], &read_fds)) {
        // client_fds[i] 可读 → 有数据
        read(client_fds[i], ...);
    }
}
```

**关键行为**：`select` 返回后，`read_fds` 已被内核**原地修改**——只有就绪的 fd 位被保留，其余清零。所以下次循环必须**重新构建** `fd_set`。

### 2.3 内核实现原理（简化）

```
用户空间              内核空间
─────────────────    ─────────────────────────────────
调用 select()    →   1. 将 fd_set 从用户空间拷贝到内核
                     2. 遍历所有 fd，注册等待队列
                     3. 进程睡眠，等待任意 fd 就绪
                     4. 某 fd 就绪 → 唤醒进程
                     5. 再次遍历所有 fd，标记就绪位
                     6. 将结果 fd_set 拷贝回用户空间
select() 返回    ←   返回就绪 fd 的数量
```

每次调用 select，发生两次全量 fd 遍历（步骤 2 和步骤 5），以及两次内存拷贝（步骤 1 和步骤 6）。

### 2.4 select 的三大致命缺陷

**缺陷 1：fd 数量上限 FD_SETSIZE = 1024**

```c
// /usr/include/x86_64-linux-gnu/bits/typesizes.h
#define __FD_SETSIZE  1024

// fd_set 本质是一个 1024 位的位图
typedef struct {
    unsigned long fds_bits[1024 / (8 * sizeof(unsigned long))];
} fd_set;
```

超过 1024 的 fd 无法被 select 监听，直接限制了并发连接数。

**缺陷 2：每次调用必须重传整个 fd_set（O(N) 拷贝）**

每次调用 select，无论有几个 fd 就绪，都要把整个 fd_set 从用户空间拷贝到内核，N 个 fd 就是 N 位的拷贝。

**缺陷 3：返回后必须线性扫描所有 fd（O(N) 扫描）**

select 只告诉你"有 k 个 fd 就绪"，但不告诉你是哪些。必须自己遍历所有 fd 逐个 `FD_ISSET` 检查，N 个 fd 就是 N 次检查。

libevent 的 `select.c` 中可以直接看到这段线性扫描：

```c
// select.c: select_dispatch() 第 193~206 行
for (j = 0; j < nfds; ++j) {
    if (++i >= nfds)
        i = 0;
    res = 0;
    if (FD_ISSET(i, sop->event_readset_out))   // 逐个检查每个 fd
        res |= EV_READ;
    if (FD_ISSET(i, sop->event_writeset_out))
        res |= EV_WRITE;
    if (res == 0)
        continue;
    evmap_io_active_(base, i, res);             // 激活就绪事件
}
```

**性能总结**：

| 操作 | 复杂度 |
|------|--------|
| 每次调用拷贝 fd_set | O(N) |
| 内核遍历注册等待队列 | O(N) |
| 用户空间扫描结果 | O(N) |
| 最大 fd 数 | 1024 |

## 3. poll：修补 fd 上限，但本质未变

### 3.1 接口

```c
#include <poll.h>

int poll(struct pollfd *fds, nfds_t nfds, int timeout);

struct pollfd {
    int   fd;       // 要监听的文件描述符
    short events;   // 感兴趣的事件（输入）：POLLIN | POLLOUT | POLLERR
    short revents;  // 实际发生的事件（输出，内核填写）
};
```

### 3.2 工作流程

```c
struct pollfd fds[MAX_FDS];
int nfds = 0;

// 注册
fds[nfds++] = (struct pollfd){server_fd, POLLIN, 0};
for (int i = 0; i < n_clients; i++)
    fds[nfds++] = (struct pollfd){client_fds[i], POLLIN, 0};

// 等待
int nready = poll(fds, nfds, 5000);  // 超时 5000ms

// 检查（仍需线性扫描）
for (int i = 0; i < nfds; i++) {
    if (fds[i].revents & POLLIN) {
        if (fds[i].fd == server_fd) accept(...);
        else read(fds[i].fd, ...);
    }
    fds[i].revents = 0;  // 重置，但 fd 和 events 无需重填（与 select 的区别）
}
```

### 3.3 poll 相比 select 的改进

| 特性 | select | poll |
|------|--------|------|
| fd 数量上限 | 1024（FD_SETSIZE） | 无硬性上限（受系统资源限制） |
| fd 注册方式 | 位图，每次重建 | pollfd 数组，events 字段复用 |
| 返回后重置 | 必须整体重建 fd_set | 只需清零 revents |

### 3.4 poll 的根本问题没有解决

poll 仍然有两个与 select 相同的根本缺陷：

1. **每次调用仍需全量拷贝**：把整个 `pollfd` 数组从用户空间传到内核
2. **返回后仍需线性扫描**：遍历所有 `pollfd`，逐个检查 `revents`

libevent 的 `poll.c` 中同样有这段扫描：

```c
// poll.c: poll_dispatch() 中的结果处理
for (i = 0; i < nfds; ++i) {
    ...
    res = 0;
    if (pop->event_set[i].revents & (POLLIN|POLLERR|POLLHUP))
        res |= EV_READ;
    if (pop->event_set[i].revents & (POLLOUT|POLLERR))
        res |= EV_WRITE;
    if (res == 0)
        continue;
    evmap_io_active_(base, pop->event_set[i].fd, res|EV_ET);
}
```

**两者的本质相同**：每次调用都是 O(N)，随着 fd 数量增长，性能线性下降。

## 4. epoll：现代 Linux 的解决方案

epoll 是 Linux 2.6 引入的高性能 IO 多路复用机制，彻底解决了 select/poll 的根本问题。

### 4.1 三个系统调用

```c
#include <sys/epoll.h>

// 1. 创建 epoll 实例，返回 epoll fd
int epoll_create1(int flags);  // flags: EPOLL_CLOEXEC
// 旧版：epoll_create(int size)，size 参数现已被忽略

// 2. 注册/修改/删除感兴趣的 fd（只需调用一次，非每次轮询）
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
// op: EPOLL_CTL_ADD / EPOLL_CTL_MOD / EPOLL_CTL_DEL
// event.events: EPOLLIN | EPOLLOUT | EPOLLERR | EPOLLHUP | EPOLLET | EPOLLONESHOT

// 3. 等待就绪事件（只返回就绪的 fd，不需要扫描）
int epoll_wait(int epfd, struct epoll_event *events,
               int maxevents, int timeout);
// 返回值：就绪事件数量
// events 数组中直接存放就绪事件（内核填写）
```

### 4.2 完整使用流程

```c
// 步骤 1：创建 epoll 实例（只做一次）
int epfd = epoll_create1(EPOLL_CLOEXEC);

// 步骤 2：注册 fd（只做一次，后续无需重复）
struct epoll_event ev;
ev.events = EPOLLIN;
ev.data.fd = server_fd;
epoll_ctl(epfd, EPOLL_CTL_ADD, server_fd, &ev);  // 注册 server_fd

for (int i = 0; i < n_clients; i++) {
    ev.events = EPOLLIN;
    ev.data.fd = client_fds[i];
    epoll_ctl(epfd, EPOLL_CTL_ADD, client_fds[i], &ev);  // 注册每个 client
}

// 步骤 3：事件循环
struct epoll_event ready_events[64];
while (1) {
    // 等待，直接返回就绪事件（不需要扫描！）
    int nready = epoll_wait(epfd, ready_events, 64, -1);

    for (int i = 0; i < nready; i++) {  // 只遍历就绪的 fd！
        int fd = ready_events[i].data.fd;
        if (fd == server_fd) {
            int client = accept(server_fd, NULL, NULL);
            ev.events = EPOLLIN;
            ev.data.fd = client;
            epoll_ctl(epfd, EPOLL_CTL_ADD, client, &ev);  // 动态注册新连接
        } else {
            char buf[1024];
            int n = read(fd, buf, sizeof(buf));
            if (n <= 0) {
                epoll_ctl(epfd, EPOLL_CTL_DEL, fd, NULL);  // 注销
                close(fd);
            } else {
                write(fd, buf, n);
            }
        }
    }
}
```

libevent 的 `epoll.c` 初始化部分对应步骤 1：

```c
// epoll.c: epoll_init() 第 174~194 行
epfd = epoll_create1(EPOLL_CLOEXEC);  // 优先使用新接口
if (epfd == INVALID_EPOLL_HANDLE) {
    epfd = epoll_create(32000);       // 降级到旧接口（size 参数被忽略）
}
epollop->events = mm_calloc(INITIAL_NEVENT, sizeof(struct epoll_event));
epollop->nevents = INITIAL_NEVENT;   // 初始 32 个事件槽，按需扩容
```

### 4.3 epoll 的内核实现原理

理解 epoll 高效的关键在于它的内部数据结构：

```
epoll 内核结构
┌─────────────────────────────────────────────────────┐
│  epoll 实例（一个 fd）                                │
│                                                     │
│  注册表（红黑树）                    就绪链表          │
│  ┌─────────────────────┐          ┌──────────────┐  │
│  │  fd1 → 等待队列节点  │          │ fd3（可读）   │  │
│  │  fd2 → 等待队列节点  │  ──触发→  │ fd7（可写）   │  │
│  │  fd3 → 等待队列节点  │          │ ...          │  │
│  │  ...                │          └──────────────┘  │
│  └─────────────────────┘                            │
└─────────────────────────────────────────────────────┘
```

- **注册表（红黑树）**：`epoll_ctl` 注册的 fd 保存在红黑树中，查找/插入/删除 O(log N)
- **就绪链表**：当某 fd 的数据就绪时，内核通过回调函数将其加入就绪链表（O(1)）
- **`epoll_wait` 只需取走就绪链表的内容**，不扫描注册表

```
调用 epoll_wait()：
  ├─ 就绪链表非空 → 直接返回，拷贝就绪事件到用户空间，O(就绪数量)
  └─ 就绪链表为空 → 进程睡眠，等待内核回调唤醒
```

### 4.4 水平触发（LT）vs 边缘触发（ET）

这是 epoll 特有的概念，select/poll 只有水平触发。

**水平触发（Level Triggered，LT）—— 默认模式**

```
fd 中有 100 字节数据：

第一次 epoll_wait → 通知（fd 可读）
用户只读了 50 字节，还剩 50 字节

第二次 epoll_wait → 还会通知（因为 fd 中仍有数据）
第三次 epoll_wait → 还会通知
...直到数据被全部读完
```

**行为**：只要 fd 中还有数据，每次 `epoll_wait` 都会报告就绪。
**优点**：编程模型简单，不会遗漏数据。
**缺点**：如果不及时读完，每次都触发，可能造成冗余唤醒。

**边缘触发（Edge Triggered，ET）—— 需显式设置 `EPOLLET`**

```
fd 中有 100 字节数据：

第一次 epoll_wait → 通知（fd 状态从"无数据"变为"有数据"）
用户只读了 50 字节，还剩 50 字节

第二次 epoll_wait → 不通知！（状态没有再次变化）
直到有新数据到来（状态再次从"无"变"有"），才触发
```

**行为**：只在 fd 状态发生变化（数据到来边沿）时触发一次。
**优点**：减少不必要的唤醒，配合非阻塞 IO 可达到最高性能。
**强制要求**：ET 模式下，收到通知后必须循环读取直到 `EAGAIN`，否则剩余数据永远不会再次触发通知。

```c
// ET 模式正确写法：必须循环读完
ev.events = EPOLLIN | EPOLLET;   // 注意加 EPOLLET
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev);

// 收到通知后：
while (1) {
    int n = read(fd, buf, sizeof(buf));
    if (n == -1 && errno == EAGAIN) break;  // 读完了
    if (n <= 0) { close(fd); break; }
    process(buf, n);
}
```

libevent 中 epoll 后端声明支持 ET 特性：

```c
// epoll.c: epollops 定义
const struct eventop epollops = {
    "epoll",
    epoll_init,
    epoll_nochangelist_add,
    epoll_nochangelist_del,
    epoll_dispatch,
    epoll_dealloc,
    1, /* need reinit */
    EV_FEATURE_ET | EV_FEATURE_O1 | EV_FEATURE_EARLY_CLOSE,  // 声明 ET 支持
    0
};
```

### 4.5 性能对比总结

| 特性 | select | poll | epoll |
|------|--------|------|-------|
| fd 数量上限 | 1024 | 无限制 | 无限制（受内存限制） |
| 注册方式 | 每次重建位图 | 每次传 pollfd 数组 | `epoll_ctl` 一次注册，永久有效 |
| 等待调用 | 全量拷贝 fd_set | 全量拷贝 pollfd 数组 | 只拷贝就绪事件（少量）|
| 结果处理 | O(N) 线性扫描 | O(N) 线性扫描 | O(就绪数量)，直接获取 |
| 内核数据结构 | 位图 | 数组 | 红黑树（注册）+ 链表（就绪）|
| 触发模式 | LT | LT | LT + ET |
| 适用场景 | 少量 fd，跨平台 | 中等 fd，无上限需求 | 大量 fd，Linux 高性能场景 |

**核心差距**：1 万个连接中有 10 个活跃时：
- select/poll：每次唤醒后扫描 10000 个 fd → O(10000)
- epoll：每次唤醒后直接处理 10 个就绪事件 → O(10)

## 5. 为什么还需要 libevent？

epoll 已经很好了，为什么还需要 libevent？

### 5.1 跨平台问题

不同操作系统提供了不同的高性能 IO 机制：

| 系统 | 推荐机制 | 对应文件 |
|------|----------|----------|
| Linux | epoll | `epoll.c` |
| macOS / BSD | kqueue | `kqueue.c` |
| Solaris | /dev/poll | `devpoll.c` |
| Solaris 10+ | event ports | `evport.c` |
| Windows | IOCP / wepoll | `wepoll.c` |
| 通用兜底 | select | `select.c` |
| 通用兜底 | poll | `poll.c` |

如果直接使用 epoll，代码无法在 macOS 上编译。libevent 提供了统一的 API，在不同平台上自动选择最优后端。

### 5.2 事件管理的复杂性

裸 epoll 只解决了"哪个 fd 就绪"的问题，但实际开发中还需要：

- **定时器**：如何在 epoll 循环中实现精确的超时回调？
- **信号处理**：如何把 SIGINT 信号转化为 epoll 可处理的事件？
- **缓冲区管理**：读写数据时如何高效管理动态大小的缓冲区？
- **连接生命周期**：accept → 读写 → 超时 → 关闭，如何优雅管理？

这些都是 libevent 在 epoll 之上提供的功能。

### 5.3 代码量对比

**裸 epoll echo server**（能力有限，代码已经较复杂）：

```c
// 约 80~100 行，且不含定时器、信号、错误处理
int epfd = epoll_create1(EPOLL_CLOEXEC);
// ... 手动管理连接数组、手动处理 EAGAIN、手动实现定时器...
```

**libevent echo server**（功能完整）：

```c
// 约 30~40 行，含完整的生命周期管理
struct event_base *base = event_base_new();
struct evconnlistener *listener = evconnlistener_new_bind(
    base, accept_cb, NULL,
    LEV_OPT_CLOSE_ON_FREE | LEV_OPT_REUSEABLE, -1, addr, addrlen);
event_base_dispatch(base);
```

### 5.4 libevent 的抽象层设计

libevent 通过 `struct eventop` 定义了后端接口：

```c
// event-internal.h
struct eventop {
    const char *name;
    void *(*init)(struct event_base *);
    int (*add)(struct event_base *, evutil_socket_t fd, short old, short events, void *);
    int (*del)(struct event_base *, evutil_socket_t fd, short old, short events, void *);
    int (*dispatch)(struct event_base *, struct timeval *);
    void (*dealloc)(struct event_base *);
    int need_reinit;
    enum event_method_feature features;
    size_t fdinfo_len;
};
```

所有后端（epoll、kqueue、select 等）都实现这个接口，上层代码完全不感知底层使用的是哪种机制。这就是为什么 libevent 能做到"一套代码，跨平台运行"。

## 6. 动手实践

### 练习 1：用 epoll 手写 echo server

目标：不使用 libevent，用裸 epoll 实现一个支持多客户端的 echo server。

```c
// echo_epoll.c 框架
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <fcntl.h>
#include <errno.h>
#include <sys/socket.h>
#include <sys/epoll.h>
#include <netinet/in.h>

#define MAX_EVENTS 64
#define PORT 9999

// 设置 fd 为非阻塞
static void set_nonblocking(int fd) {
    int flags = fcntl(fd, F_GETFL, 0);
    fcntl(fd, F_SETFL, flags | O_NONBLOCK);
}

int main(void) {
    // 1. 创建 server socket
    int server_fd = socket(AF_INET, SOCK_STREAM, 0);
    int opt = 1;
    setsockopt(server_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

    struct sockaddr_in addr = {
        .sin_family = AF_INET,
        .sin_addr.s_addr = INADDR_ANY,
        .sin_port = htons(PORT),
    };
    bind(server_fd, (struct sockaddr *)&addr, sizeof(addr));
    listen(server_fd, 128);
    set_nonblocking(server_fd);

    // 2. 创建 epoll 实例
    int epfd = epoll_create1(EPOLL_CLOEXEC);

    // 3. 注册 server_fd
    struct epoll_event ev = { .events = EPOLLIN, .data.fd = server_fd };
    epoll_ctl(epfd, EPOLL_CTL_ADD, server_fd, &ev);

    printf("Echo server listening on port %d\n", PORT);

    // 4. 事件循环
    struct epoll_event events[MAX_EVENTS];
    while (1) {
        int nready = epoll_wait(epfd, events, MAX_EVENTS, -1);
        for (int i = 0; i < nready; i++) {
            int fd = events[i].data.fd;

            if (fd == server_fd) {
                // 新连接
                int client = accept(server_fd, NULL, NULL);
                set_nonblocking(client);
                ev.events = EPOLLIN;
                ev.data.fd = client;
                epoll_ctl(epfd, EPOLL_CTL_ADD, client, &ev);
                printf("New connection: fd=%d\n", client);
            } else if (events[i].events & EPOLLIN) {
                // 有数据可读
                char buf[1024];
                int n = read(fd, buf, sizeof(buf));
                if (n <= 0) {
                    epoll_ctl(epfd, EPOLL_CTL_DEL, fd, NULL);
                    close(fd);
                    printf("Connection closed: fd=%d\n", fd);
                } else {
                    write(fd, buf, n);  // echo 回去
                }
            }
        }
    }
    return 0;
}
```

## 7. 本课小结

| 机制 | 解决的问题 | 遗留的问题 |
|------|-----------|-----------|
| select | 单线程监听多 fd | fd 上限 1024；O(N) 拷贝和扫描 |
| poll | 突破 fd 数量限制 | 仍然 O(N) 拷贝和扫描 |
| epoll | O(1) 获取就绪事件；无 fd 上限 | 仅限 Linux；不含定时器/信号等抽象 |
| libevent | 跨平台统一封装；提供定时器/缓冲/HTTP 等高级功能 | — |

**三句话总结**：

1. select 用位图标记 fd，最多 1024 个，每次 O(N) 拷贝和扫描
2. poll 改为 pollfd 数组，突破数量限制，但仍是 O(N)
3. epoll 用红黑树注册、就绪链表返回，O(就绪数量)，是高性能服务器的标配
